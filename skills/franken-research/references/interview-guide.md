# Interview Guide

Detailed methodology for the interview mode of franken-research. This mode explores all repositories in the workspace, then conducts a multi-round Q&A to understand what the user is building, ultimately generating a tailored research prompt.

## Step 1: Auto-Detect Repositories

Scan the workspace root for directories that look like repositories. A directory is a repo candidate if it contains any of:

- `.git/` directory
- `README.md` or `README.rst`
- Language markers: `Cargo.toml`, `package.json`, `go.mod`, `pyproject.toml`, `setup.py`, `requirements.txt`, `pom.xml`, `build.gradle`, `CMakeLists.txt`, `Makefile`, `mix.exs`, `Gemfile`, `.csproj`

Exclude: `node_modules`, `.git`, `target`, `dist`, `build`, `__pycache__`, `.venv`, `vendor`, `research_docs`, hidden directories (starting with `.`).

Report the detected repos to the user before proceeding. Ask if any should be excluded or if any were missed.

## Step 2: Explore All Repos

Launch explore agents at medium depth. Goal: understand what each repo IS and what feature categories it covers — enough to ask informed questions, not a full deep dive.

**Agent budget:**
- ≤4 repos: 2 explore agents per repo (one for architecture/core, one for features/extensions)
- 5+ repos: 1 explore agent per repo
- Launch in batches of 3 (parallel)

**Each explore agent should cover:**
- Read README.md and any project documentation (CLAUDE.md, AGENTS.md, CONTRIBUTING.md, .cursorrules)
- Map top-level directory structure and module boundaries
- Identify the main execution path (entrypoint → core loop → key abstractions)
- Catalog 3-7 major feature categories (e.g., "memory system", "plugin architecture", "multi-agent coordination")
- Note the primary language, approximate size (file count or LOC), and maturity level
- Identify what seems genuinely novel or unusually thorough vs. table stakes

## Step 3: Write Landscape Summary

Write `research_docs/landscape.md` with this format:

```markdown
# Repository Landscape

Generated by franken-research interview mode.

## Repositories

### {repo-name}
- **Path**: `{absolute-path}`
- **Language**: {primary language}
- **Size**: ~{approximate size}
- **Purpose**: {1-2 sentence description of what it does}
- **Key feature categories**: {comma-separated list of major feature areas}
- **Notable**: {1 sentence on what seems most distinctive or well-implemented}

[Repeat for each repo]

## Cross-Cutting Observations

- {2-5 bullet points on patterns that appear across multiple repos}
- {Note which feature areas have strong coverage vs. gaps}
```

## Step 4: Multi-Round Q&A

Informed by the exploration results, conduct a multi-round interview to understand what the user is building. Use AskUserQuestion for structured choices and direct questions for open-ended topics.

### Question Categories

Work through these categories across multiple rounds. Adapt based on what's relevant — not every project needs every category. Ask 3-4 questions per round.

**Round 1: Project Identity**
- What are you building? (form factor: CLI, library, web service, framework, etc.)
- What language/stack?
- Who is the target user? (developers, end users, both)
- What's the positioning? (general-purpose, domain-specific, opinionated, flexible)

**Round 2: Capabilities and Priorities**
Based on the feature categories discovered across repos, ask which capabilities are:
- Essential (day-one, non-negotiable)
- Important (should have, but can iterate)
- Nice-to-have (only if well-designed and low-cost)
- Not relevant (skip entirely)

Example: "I found that 5 of your 8 repos have memory/learning systems. Three different approaches exist: [approach A], [approach B], [approach C]. Which resonates most with what you're building?"

**Round 3: Open Technical Decisions**
For areas where multiple repos take different approaches, surface the trade-offs:
- "For X, repo A does [approach], repo B does [approach]. Do you have a preference, or should research inform this?"
- "Several repos support Y but implement it very differently. What matters most to you: [dimension A] vs [dimension B]?"

**Round 4+: Scope and Refinement**
- Evaluation criteria: How should features be ranked? (all dimensions equally, or weight some higher?)
- What counts as "table stakes" vs. "genuinely distinctive" for this project?
- Any constraints? (performance budgets, compatibility requirements, deployment targets)
- Anything specific you've already decided on that research shouldn't revisit?

### Convergence

Continue asking rounds until the user signals they're ready:
- "Generate the prompt"
- "That's enough"
- "Let's go"
- "I'm ready"
- Any clear signal that the interview phase is complete

Do NOT stop early on your own. The user controls when to move to prompt generation.

## Step 5: Generate Research Prompt

Use `references/prompt-template.md` as the skeleton. Populate it with:

1. **"What We're Building" section** — transform Q&A answers into a structured project description:
   - Project name and form factor
   - Defining characteristics (from Round 1 answers)
   - Essential capabilities with brief descriptions (from Round 2)
   - Open technical decisions (from Round 3)
   - Evaluation guidance (from Round 4+)

2. **Repo table** — populate from landscape.md data:
   ```
   | Repo | Path | Description |
   |------|------|-------------|
   | {name} | `{path}` | {one-line description from landscape} |
   ```

3. **Output sections** — use the 6 recommended core sections from `references/research-execution.md`, adjusted if the interview revealed project-specific needs

4. **Quality standards** — include the standard quality standards block

Write the result to `research_docs/research_prompt.md`.

Inform the user that the prompt is ready and explain how to use it: feed it to an agent along with the target repo name to produce a findings document.
