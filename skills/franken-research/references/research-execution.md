# Research Execution: 4-Phase Deep Dive

Methodology for the research mode of franken-research. Follow these 4 phases when deep-diving a single repository to extract its most impactful features.

## Prerequisites

- `research_docs/research_prompt.md` must exist (generated by interview mode)
- Read the research prompt first — it contains the project description, repo table, output template, and quality standards

## Phase 1: Orientation

Before launching any deep exploration, ground yourself:

1. **Read existing findings docs** in `research_docs/` — understand what features have already been identified in other repos. This is critical for differentiation: focus on what makes THIS repo genuinely different.

2. **Read the target repo's README.md** and any project documentation (CLAUDE.md, AGENTS.md, CONTRIBUTING.md, .cursorrules). These tell you what the project claims to be and how it's organized.

3. **Map the directory structure** — understand the top-level layout, where the core source code lives, what the module/package boundaries are.

4. Based on steps 1-3, identify 3 distinct exploration domains that cover the full surface area of this repo. Each domain should be explored by a separate agent in parallel.

## Phase 2: Deep Parallel Exploration

Launch up to 3 explore agents in parallel, each with a distinct focus area. Common domain splits (adapt based on repo structure):

- **Agent A: Core execution architecture** — How the main abstractions are defined, configured, and executed. The main loop, tool calling, streaming, structured output, state management. Multi-agent coordination if present.

- **Agent B: Intelligence and context layer** — Knowledge/RAG, memory, learning, reasoning, context management/compression, guardrails, evaluations. How the system gets smarter over time or manages long contexts.

- **Agent C: Infrastructure and extensions** — Abstraction layers, tool system breadth, persistence/storage, serving/API layer, observability/tracing, configuration, plugin/extension system, CLI experience.

Each explore agent should:
- Read actual source files, not just READMEs
- Trace the main execution path through the code
- Identify the key classes/structs/functions and their relationships
- Read 2-3 cookbook/example files to see how the API is actually used
- Note file paths and line numbers for key implementations
- Focus on what seems novel, well-designed, or unusually thorough
- Investigate performance-relevant patterns: streaming/buffering semantics, prompt caching, concurrency/parallelism, batching, and other optimizations that affect real-world efficiency

## Phase 3: Synthesis and Differentiation

After exploration results return, evaluate each feature or pattern:

1. **Is this genuinely novel?** Check against existing findings docs and the "What We're Building" section. If another repo already has a stronger version, it's not this repo's distinctive contribution. If this repo has a unique approach no other repo uses, that IS novel.

2. **Is this well-implemented?** A half-baked feature is less valuable than one that's deeply thought through. Look for sophistication: edge case handling, extensibility, clean abstractions.

3. **Would this make our project better?** Evaluate against the project's stated priorities and evaluation criteria from the research prompt.

4. **Is this table stakes or genuinely distinctive?** Common features (tool systems, streaming, config loading) are table stakes. Document them in Feature Analysis but do NOT promote to the "adopt" section unless this repo's implementation is genuinely superior.

Optional: briefly explore 1-2 other repos to confirm a feature you think is novel isn't already present elsewhere.

## Phase 4: Write the Research Document

Write `research_docs/{repo}_findings.md` following the output template from the research prompt.

### Recommended Core Sections

These 6 sections are recommended as the default structure. The research prompt may adjust these based on the project's needs.

#### 1. Scope and Method
- What was explored, how, which agents were launched and what each covered
- Primary sources reviewed (README, docs, key source files with paths)

#### 2. README Alignment: What Is Unique About This Project
- What does the README claim this project is?
- For each major claim (3-5 claims), verify against actual code with file:line references
- What is the project's genuine identity/positioning? (2-3 sentences)

#### 3. Architecture Overview
- High-level architecture (3-5 sentences)
- Module/directory map with purposes
- Key abstractions and how they relate

#### 4. Feature Analysis

Use this exact format for every feature:

```
### N. Feature Name (subsystem path)
**What it is**: One sentence.
**Key files**: file paths and approximate sizes
**How it works**: Implementation details with file:line code references. Describe the PATTERN and ALGORITHM, not a line-by-line walkthrough.
**Notable details**: Parameters, edge cases, design decisions that matter
```

Be thorough — cover everything found, including table-stakes features. The next section filters for what's genuinely novel.

#### 5. What Our Project Should Adopt From {REPO}

The most valuable section. Each item must have enough detail that a developer could implement it. Do NOT pad with table-stakes features.

```
### N. Feature Name (IMPACT LEVEL)

**The idea**: What is it in one sentence.

**Why this matters**: Why this is valuable for our project. What problem does it solve. What happens if we don't have this.

**How it works** (language-agnostic pseudocode/pattern description):
- Core algorithm, data structures, and state transitions
- Key abstractions and their relationships
- Configuration model and API surface
- Enough detail to implement without reading the original source

**Source**: File paths with approximate sizes (for reference, not for copying)
```

Rank by impact: HIGHEST, VERY HIGH, HIGH, MEDIUM-HIGH, MEDIUM.

#### 6. Summary

One-line-per-feature list of this repo's gifts to the project, in order of impact.

## Quality Standards

These are hard requirements. Documents that violate these should be redone.

- **Self-contained**: The document must stand alone. Do not reference other findings docs in the body.

- **Code-grounded**: Every claim must trace to a file path. Use `path/to/file.ext` (approximate size) format. Do NOT copy large blocks of source code.

- **Pseudocode over source code**: Describe PATTERNS using language-agnostic pseudocode or algorithmic descriptions. Do NOT paste raw source code from the repo. The output should be re-implementable in any language.

- **Honest about impact**: Not every feature is high impact. A short "adopt" list with genuinely novel items is far more valuable than a padded list.

- **Implementation-focused**: Each "adopt" item needs algorithms, data structures, state machines, and integration points — not just the idea.

- **No comparison sections**: Do NOT include tables comparing repos, "weaknesses", "risks/tradeoffs", "practical blueprint", or "what to simplify" sections. Each doc is purely about what THIS repo contributes. Cross-repo synthesis happens separately.

- **Strict section structure**: Use ONLY the sections specified in the research prompt. Do not add Table of Contents, Appendix, File Index, Reference Index, Blueprint, or any other sections.
